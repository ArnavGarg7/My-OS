# Performance Review — v1.0.0

Verification that the 1.0.0 release meets the performance bar set at the Phase 4.5 freeze and that
Phase 6 introduced no regression. **Result: PASS.**

## Method

Two authoritative baselines were established at the freeze and are the reference for this release:

- **Bundle** — [bundle.md](bundle.md), generated by `scripts/bundle-report.mjs` from the Next.js
  **production** build manifest: shared baseline **342.8 kB**, route-specific ~555–850 kB.
- **Core derivations** — [baseline.md](baseline.md), generated by `scripts/benchmark.mjs`: every pure
  read-model derivation completes in **well under a millisecond** (e.g. dashboard 30.8 µs mean).

> Note: running `scripts/bundle-report.mjs` without a fresh production `.next` over-reports (it falls
> back to source size). The figures above are from a real production build and remain the reference.
> A production build was **not** re-run in this review to avoid the documented `.next` corruption risk
> against a live dev server (see [guides/debugging.md](../guides/debugging.md)).

## Bundle sizes — no regression from Phase 6

Phase 6 added **five routes** (`/signals`, `/prediction`, `/autopilot`, `/connectors`, `/adaptation`),
each a Tabs-based "Center" built from the existing `@myos/ui` component library. Crucially, **Phase 6
added no new runtime dependencies** — the web app's dependency set is unchanged (`@clerk/nextjs`,
tRPC, react-query, drizzle, lucide-react, next, react, zustand, zod, and the `@myos/*` workspace
packages). Therefore the new routes fall into the same route-specific band as their peers; the shared
baseline is unchanged because no new shared dependency was introduced.

## Lazy loading / route splitting

- **Route splitting** — every page under `app/(shell)/*` is its own route with its own code-split
  chunk; the shell shares the framework baseline only.
- **Lazy heavy client component** — the Intelligence Wheel is loaded via `next/dynamic` with
  `ssr: false` (`components/intelligence/Dashboard.tsx`), keeping the only heavy client-only
  visualization out of the server render and off other routes' first load.
- **Server components by default** — Centers render server-side; interactivity is isolated to
  `"use client"` leaves.

## Database indexes & query efficiency

- **120 indexes** declared across the schema files, including the Phase 6 additions (signals by
  status/category, predictions by kind, `autopilot_*` by proposal, `connector_*` by account/kind,
  adaptation by key/category). Every hot lookup path (by account, by key, by kind, by status) is
  indexed.
- **Repositories select narrow columns and bound results** (`.limit(...)`, ordered by an indexed
  column). Derived read models are computed in pure core from already-fetched rows — no N+1.

## Render / hydration / rerenders

- **37 components** use `useMemo` / `useCallback` / `memo` to stabilize expensive derivations and
  callbacks.
- Engine cycles (signals, prediction, autopilot, adaptation) run **on-demand** server-side and return
  already-derived, ranked/banded read models — the client renders plain data, no client-side
  computation of intelligence.

## Conclusion

The 1.0.0 release holds the Phase 4.5 performance baseline. Phase 6 is additive with no new
dependencies, no shared-baseline growth, and the same per-route pattern, so bundle and derivation
budgets are preserved. Follow-up (non-blocking): capture a fresh production `bundle.md` on the release
machine to record exact 1.0.0 first-load numbers for the five new routes.
